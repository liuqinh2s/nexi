编程绕不开的一个话题：耦合。

耦合其实是个中性词，代码模块之间要相互协作完成任务，就必定要耦合。**耦合即：一方对另一方的假设**。你不可能对协作对象不做任何假设，所以有些耦合是必须的。我们要解决的是非必要的、不良的耦合。

> 如无必要，不要使代码耦合。过度耦合会加速代码腐烂，比如难加新功能、难修 bug。

对另一方做尽可能少的假设只是手段之一，而且这种手段会让代码变得更加抽象。另一种手段是制定标准和建立知识库，大家用共同的标准和共有的知识沟通。

# 怎么写出低耦合的代码

1. 模块保持单向依赖关系。
2. 接口保持正交性和紧凑性。
3. 提供原子化底层 API，再提供面向业务的胶水层 API。
4. 既做防御式编程，也提供模块和接口的最佳实践。

## 单向依赖

模块之间尽可能保持单向依赖，形成树状结构，而不是难以梳理的网状结构。

## 正交性

正交性是指一个模块提供的 API 中，多个方法之间是否存在重复功能。

> 想象一下，调节电视声音的时候，亮度也会跟着一起变化。

通常正交性高的模块更稳定，不会因为上层业务变化被迫修改。底层 API 应该只实现正交的机制；如果感觉拆得太细不方便使用，可以在它们之外包装 Helper 或 Utility 组成胶水层。

## 紧凑性

紧凑性是指一个模块提供的公有方法总数和每个方法的参数都应该较少。《Unix 编程艺术》提到，一个模块的方法数量过多后会变得难以理解。

单向依赖、正交性和紧凑性都很务实，也有相对客观的方法衡量。

## 不同级别之间的耦合

软件由不同级别的概念层组成。方法、类、包、协议、语言、数据流、数据库和业务层中，都存在不同形式的耦合。

## 总结

耦合的本质是假设。假设越多，被打破的概率越高，软件的可靠性就越低。低耦合代表软件在不同级别上只依赖真正需要的内容，使自身修改不至于给其他系统造成非必要影响，反之亦然。

开发过程中应该持续管理这些耦合，并经常考虑：如果某个假设被打破，会给系统带来什么风险？
